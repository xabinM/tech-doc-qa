# Harness 작업 수행 절차

## `add rule {이름}` 수행 절차
1. `.claude/rules/{이름}.md` 생성 (아래 템플릿 사용)
2. 가장 관련 있는 `*/CLAUDE.md` 파일에 `@../.claude/rules/{이름}.md` 추가
3. 추가된 위치 보고

**rule 파일 템플릿:**
```markdown
# {제목} 규칙

- 규칙 항목 1
- 규칙 항목 2
```

---

## `add agent {이름}` 수행 절차
1. 기존 agents 목록을 읽어 역할 중복 여부 확인, 중복이면 경고 후 확인
2. `.claude/agents/{이름}.md` 생성 (아래 템플릿 사용)
3. description은 "언제 사용하는지" 기준으로 구체적으로 작성

**agent 파일 템플릿:**
```markdown
---
name: {이름}
description: {트리거 조건을 포함한 구체적 설명. 언제, 무엇을 위해 사용하는지 명시.}
---

너는 이 프로젝트의 {역할}을 담당하는 전문가다.

# 프로젝트 컨텍스트
{관련 기술 스택, 파일 경로 등}

# 작업 원칙
{구체적인 행동 지침}

# 작업 순서
1. ...
2. ...
```

---

## `add command {이름}` 수행 절차
1. `.claude/commands/{이름}.md` 생성
2. 첫 줄에 `$ARGUMENTS` 처리 방식 명시
3. 인자 없을 때 기본 동작 명시

**command 파일 템플릿:**
```markdown
$ARGUMENTS 에 대해 {동작}한다.

**인자 없이 실행하면:** {기본 동작}
**인자가 있으면:** {인자별 동작}
```

---

## `split {파일경로}` 수행 절차
1. 대상 파일 읽기
2. 논리적 단위(주제, 계층, 도메인) 기준으로 분리 계획 제시
3. **반드시 멈추고 사용자 확인** — 생성될 파일 목록과 삭제될 파일 명시
4. 확인 후: 분리된 신규 파일 먼저 생성
5. 원본이 CLAUDE.md인 경우: 분리된 섹션을 @import로 교체 (삭제 아님)
6. 원본이 rules/agents 파일인 경우: **신규 파일 생성 완료 확인 후** 원본 삭제
   - 삭제 전 "분리된 파일이 정상적으로 생성됐습니다. 원본 {경로}를 삭제해도 될까요?" 재확인

---

## `sync` 수행 절차
1. `git log --oneline -20` 로 최근 커밋 파악
2. `git diff main...HEAD --name-only` 로 변경된 파일 목록 확인
3. 변경된 파일 읽어 새로운 패턴/의존성/도메인 파악
4. 현재 harness 파일들과 비교해 반영 안 된 항목 목록화
5. 각 항목에 대해 "어떤 파일을 어떻게 수정/추가할지" 제안
6. 사용자 승인 후 실행

---

## `update {파일경로}` 수행 절차
1. 대상 harness 파일 읽기
2. 관련 코드 파일 읽기 (agent라면 다루는 도메인 코드, rule이라면 관련 계층 코드)
3. 현재 코드와 harness 내용 간 불일치 항목 목록화
4. 수정 제안 후 사용자 확인
5. 확인 후 업데이트
